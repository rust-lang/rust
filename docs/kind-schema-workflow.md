# Kind Schema Workflow

This document describes how Kind schema definitions in ThingOS are managed,
how generated code is kept in sync, and how drift is detected and resolved.

## Overview

ThingOS uses `kindc` (the Kind schema compiler) to generate canonical Rust types
from `.kind` schema definition files. These generated types serve as the single
source of truth for system boundaries, wire formats, and kernel-facing interfaces.

The canonical schema files live in `tools/kindc/kinds/` and the generated Rust
output is checked into `tools/kindc/fixtures/generated/mod.rs`.

## Schema Change Workflow

### Making changes to Kind definitions

1. Edit the relevant `.kind` file(s) in `tools/kindc/kinds/`.
2. Run `just kindc-gen` to regenerate the Rust output:

   ```bash
   just kindc-gen
   ```

3. Review the diff in `tools/kindc/fixtures/generated/mod.rs` to confirm the
   generated code matches your expectations.
4. Commit both the `.kind` source changes and the updated generated file together.

### Checking for drift locally

To verify that the checked-in generated code is up to date with the schema sources:

```bash
just kindc-check
```

This command regenerates the output and uses `git diff --exit-code` to detect any
mismatch. It exits with a non-zero status and prints an error if drift is found.

## Generated Files Are Read-Only

The file `tools/kindc/fixtures/generated/mod.rs` is marked with a header comment:

```rust
// @generated by kindc — DO NOT EDIT.
```

**Do not manually edit this file.** Manual changes will:

- Be overwritten the next time `just kindc-gen` runs.
- Cause the CI drift check to fail.
- Break the Kind-first architecture guarantee.

If the generated output does not match what you need, the correct fix is to update
the `.kind` schema definition and regenerate.

## CI Drift Detection

A dedicated GitHub Actions workflow (`.github/workflows/kindc-drift.yml`) runs on
every pull request and push to `main` that touches:

- `tools/kindc/kinds/**` — schema source files
- `tools/kindc/src/**` — the compiler itself
- `tools/kindc/fixtures/generated/**` — the generated output
- `tools/kindc/Cargo.toml` — compiler dependencies

The workflow regenerates the Rust output from the current `.kind` sources and
fails if the result differs from the checked-in file. A failure means the
generated file is either stale or was manually modified.

### Resolving a CI drift failure

1. Pull the branch and run:

   ```bash
   just kindc-gen
   ```

2. Commit the updated `tools/kindc/fixtures/generated/mod.rs`.
3. Push the updated commit.

## Principles

- **Kind definitions are the source of truth.** The `.kind` files define the
  canonical shape of all system types. Do not define equivalent structs by hand
  in kernel or userspace code.
- **Generated code is a build artifact checked in for convenience.** It is
  committed so that downstream crates can depend on it without running `kindc`
  at build time, but it must always reflect the current schema.
- **Drift is a build-time or CI failure.** Any divergence between the schema and
  the generated output must be resolved before a branch can be merged.
