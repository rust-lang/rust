# v0.5 Tasking Arch Boundary

## Goal
Establish a strict boundary between arch-neutral kernel scheduling and arch-specific execution mechanics.

## Kernel Ownership
- **Task Management**: Task IDs, states (Runnable, Running, etc.), priority.
- **Scheduling**: Run queues, picking the next task, policy.
- **Control Flow**: `yield_now`, `sleep`, `spawn` (higher level).
- **Bookkeeping**: Task total count, current task ID.

## BootRuntime Ownership
- **Context management**: Saving and restoring CPU state (registers, stack pointer).
- **Paging / Address Spaces**: Creating page tables, switching address spaces (satp, CR3).
- **Entry / Return**: Building the initial frame for user/kernel tasks, performing the final jump.
- **Mapping**: Handling ELF segment mapping requests with arch-specific flags.

## The Arch Hook Surface
The kernel interacts with the arch via the `BootTasking` trait (part of `BootRuntime`):

- `init_kernel_context(entry, kstack_top)`: Build a `TaskContext`.
- `init_user_context(spec)`: Build a `TaskContext` and an `AddressSpace`.
- `switch(from, to)`: The core context switch.

## Invariants
1. The kernel MUST NOT import any symbols from `crate::arch::imp` in scheduling paths.
2. The kernel MUST NOT know the layout of `TaskContext`.
3. Architecture-specific page table formats (PML4, satp) MUST NOT be visible to the kernel.
